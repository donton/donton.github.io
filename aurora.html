Aurora

Goal

Design and develop the entire front-end for Aurora, a guitar-teaching tool

Process

Concept development, research user needs, lo-fidelity prototype, testing, hi-fidelity prototype, UI development, brand development

My Role

UX Design, Front-end Development

Introduction

Aurora is an onboard guitar-teaching tool aimed at beginners learning to play guitar. The device is comprised of light and sensors that are placed on the neck of a guitar underneath the strings as well as a microcontroller and touchscreen running the entire system. The lights guide users to learn guitar techniques as well as songs.

I was given the opportunity to work on Aurora in college. Because our team was small, I was able to be in charge of Aurora's entire front end, from design to development. My primary task was to develop the UI for the onboard touchscreen. Towards the end, we were able to take it step further, and I created a simple website for Aurora. I knew this wouldn't be an easy task as designing for an atypical device type definitely comes with its challenges.

Discovery & Research

My team developed Aurora starting from its concept. We discussed implementation and hardware solutions and determined that an onboard touchscreen would work best for our UX needs. This section outlines the discovery and research I conducted for the UX.

Goals

  • Be able to select different modes and songs
  • Display relevant information onscreen while playing
  • Include enough information in a small screen for a user to understand what's going on
  • Not be a nuisance to the user during the guitar-learning experience

Functionality

As a team, we discussed what we wanted Aurora to be. We defined our scope with the project and determined features needed. We also took a look at our competition to figure out how we can create a better guitar-teaching tool.

Features

Lessons – teaches a user certain guitar techniques
Learn – teaches a user how to play songs step-by-step, waits for a user before moving on
Play – teaches a user a song but doesn't wait for them, plays at normal pace
Record – allows a user to record their own song to learn and play later

Competitive Advantage

After combing through similar devices, we found that they all lacked one thing, a feedback system. This was something we added on to our device with a system of sensors that detect if a user is placing their fingers in the correct spots. This meant the UI had to be responsive and could wait for the user.

Conceptual Design

Because Aurora is supposed to be a supplementary tool, it was especially important that the touchscreen not intrude on the user's learning experience. I defined the UI around its top three functionality components being sure not to include anything unnecessary.

Three Main Functionality Components

Lessons & Learn

Users select lessons or songs to learn

Lights indicate finger placement, Aurora waits for the user to correctly place their fingers and strum before moving on

The UI displays the user's current progress through the lesson or song

Play

User select songs to play

Lights indicate finger placement but goes at the normal song beat

The UI displays the user's current progress through the song

Displays the user's accuracy at the end

Record

User selects mode to record song

The UI walks the user through recording a song note by note

The user can then learn or play the song later

The following sketches represent the UI flow of the entire system along with wireframes of the individual screens. I had to be cognizant of the fact that I was designing for a 3.5" touchscreen, a screen much smaller than today's smartphones.

Testing

After sketching out my designs, I conducted internal testing to make sure all of our features were efficiently part of the interface. Some suggestions I received were to try to include less text (because of screen size) as well as include some automatic redirects, like directing a user to play a song directly after they finished the lesson for it. After validation, I was able to move on to create hi-fidelity mockups.

Designing for such a unique device requires another level of empathy to understand and cater to the user who will be interacting with a new and unfamiliar experience.

Final Design

Touchscreen UI

This is my final design for the screens arranged in a hierarchal order. Users start by selecting a mode, then a lesson or song, they play it, then it takes them to the end screen. I designed the UI to be a very nonintrusive experience to the user. They can get to the function they want, play it without too much distracting them onscreen, and move on. Most of the functions are designed similarly so the learning curve to Aurora isn't high; users can pick it up instantly.

Some Development Challenges

As the designer as well as the developer, I was able to bridge the gap between the design and code. I ran into many issues developing on a niche touchscreen display that affected the design. I had to make some compromises. For example, the display couldn't efficiently render images, everything had to be drawn on the screen from scratch. This meant we couldn't include our logo or any other images, this was where color and typography really did its work to convey our brand image. The display was also very slow! Compared to our devices today, they screen took forever to draw any figure. This was intrusive to the user experience. I had to be especially mindful and created simplistic designs that wouldn't interfere with anyone's experience.

Aurora Website

Towards the end of our project, we had some time to implement some more features. Something we thought would be helpful was a website for users to download more songs directly to their device. The bulk of this site would be the actual browse-for-songs functionality, so that's what I designed below. Songs are organized by collections where users can buy collections as a whole or individual songs.

Brand Design

I wanted Aurora to be a fun experience. The colors represent an actual aurora while the logo is reminiscent of a guitar neck.

Final Thoughts

After working on Aurora for several months, we ended up with a product we were proud of. The entire front-end of the system turned out beautifully. Being the designer and front-end developer allowed me to follow the design and development lifecycle in its entirety and see where gaps typically fall when working between the two.

Designing for a unique device type really showed me that this process isn't easy. We have many tools that we take for granted today that speed up the design and development process that I wasn't able to utilize. The touchscreen was outdated, and coding on the screen was no easy task. It was very time-consuming for what you got out of it. Something we would do differently would be to use a smartphone app instead of an onboard touchscreen display. That would solve many problems and make Aurora more future-proof.

Given the constraints of our product as well as the technology available, my most challenging task was to create a nonintrusive yet fun learning experience for the user. I believe that the simplicity of my design coupled with the device's engaging features highlights the uniqueness of the device and make it an enriching experience for the user.
